Reporting Platform Failover Mechanism

TL;DR

Add automatic failover to the bureau reporting pipeline so that a primary node failure triggers seamless cutover to a secondary node, eliminating the single point of failure causing intermittent outages and achieving 99.9%+ uptime.

Problem

The bureau reporting platform runs on a single primary node with no automated recovery path. When that node goes down (due to hardware fault, deployment failure, or network partition) report submissions stall until an engineer manually restores service. This has caused missed bureau submission windows on three occasions in the past year, creating compliance risk and client-facing SLA breaches. Mean time to recovery (MTTR) is currently 45–90 minutes, all of it manual.

Goals

✓Automatic failover to secondary node within 60 seconds of primary failure
✓Zero manual intervention required for failover to complete
✓No duplicate bureau submissions during or after failover
✓On-call alert fires within 2 minutes of primary failure
✓Runbook and playbook documented for ops team

Non-goals

×Active-active multi-region setup (out of scope for this phase)
×Automatic failback: ops team initiates failback manually after investigation
×Changes to the report processing logic or data model
P0 Health check endpoint on primary node, polled every 15 seconds by a watchdog service. Three consecutive failures trigger failover.
P0 Secondary node runs in warm-standby: data in sync, service ready but not accepting traffic. Traffic switches via load balancer rule update.
P0 Idempotency key on all report submissions. Any in-flight report at time of failover is retried on secondary; duplicate check prevents double submission.
P1 PagerDuty alert fired on failover event with: timestamp, failure reason, affected reports count, secondary node status.
P1 Failover event logged to audit trail with full context, required for compliance review.
P2 Dashboard widget showing primary/secondary node health status, updated every 30 seconds.

Success metrics

Platform uptime ≥ 99.9% over rolling 30 days
Failover completes in < 60s on 100% of triggered events
Zero duplicate bureau submissions post-launch
MTTR drops from ~67 min average to < 5 min

Open questions

How long should the secondary stay promoted before ops must manually evaluate failback? Define SLA.
Should bureau partners be notified automatically on failover, or only if submission window is missed?
What constitutes a healthy primary before failback is permitted? Need agreed criteria.
