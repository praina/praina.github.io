Tractable · 2025–Present

Enterprise Platform Migration

Leading end-to-end migration of enterprise clients onto a new AI-powered platform architecture, from discovery and scoping through testing and delivery, with full feature parity and zero disruption to live operations.

Context

Tractable builds AI-powered claims automation for the insurance industry. As the product evolved, legacy enterprise clients needed to move onto a new platform architecture that unlocked modern AI capabilities, better integration patterns, and improved configurability, without any break in the real-time operations these clients depend on.

Problem

Enterprise insurance clients run 24/7 claims operations. A migration that touches their integration layer, data flows, or configuration model carries significant risk: downtime or feature regression can cascade into financial and reputational damage for both the client and Tractable. The challenge wasn't just technical: it required aligning multiple enterprise clients, an engineering team, and internal ops, each with different timelines, risk tolerances, and feature dependencies.

Approach

01

Deep discovery: map the dependency graph before touching anything

Worked with engineering and client ops to document every integration touchpoint, configuration dependency, and data flow for each enterprise client. Built a feature-parity checklist so "done" had a clear, agreed definition before a single line of code changed.

02

Scoped a phased migration plan with explicit rollback gates

Designed a client-by-client migration sequence ordered by integration complexity. Each phase had a defined rollback trigger so we could abort without impact. Brought the plan to clients early: no surprises at go-live.

03

Ran testing in parallel environments before any production cutover

Coordinated shadow-mode testing: new platform ran alongside legacy, outputs compared automatically. Gaps found in testing, not in production. Translated test findings into sprint-ready bug reports and requirement clarifications.

04

Managed stakeholder alignment through the inevitable pivots

Client timelines shifted. Engineering uncovered unexpected dependencies mid-sprint. Maintained a shared status dashboard and weekly touchpoints with client ops and engineering leads, so re-scoping decisions were made collaboratively, not reactively.

What I'd carry forward

"Definition of done" must include client sign-off on feature parity, not just engineering sign-off. These are different gates and conflating them is where migrations slip.

The rollback gate isn't pessimism: it's the thing that lets you move faster. Every stakeholder relaxes when they know abort is safe.
